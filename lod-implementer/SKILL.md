# LOD Implementer

## What is LOD?

**LLM-Oriented Design (LOD)** is a set of architectural patterns that optimize code for AI comprehension. LLMs have a finite intelligence window -- large files, mixed responsibilities, and implicit interfaces increase hallucination risk and waste context. LOD enforces small files (<800 LOC), single responsibilities, explicit calling specs on every module, and clean separation of deterministic (sealed, tested) vs probabilistic (flexible, LLM-domain) logic. LOD consists of 7 Hard Rules (always enforced) and 9 Design Patterns (situationally applied).

## Purpose

Execute an LOD-constrained feature plan phase by phase. This skill reads a plan document, implements each phase following TDD Red-Green-Refactor, validates LOD Hard Rules at every quality gate, and updates the plan with progress.

**Use this skill** when you have a plan at `docs/plans/PLAN_<name>.md` generated by lod-planner (or any plan following the LOD phase structure).

**Do not use** when there is no plan document, requirements are unclear (use lod-planner first), or the task is a quick fix that doesn't need phased execution.

---

## LOD Hard Rules Reference

These 7 rules must be validated at every quality gate. Violation = stop and fix before proceeding.

1. **Max 800 LOC per file.**
2. **One file, one responsibility.**
3. **Pure functions over methods.** No `self`/`this` unless instance state is needed.
4. **Flat over deep.** No inheritance >1 level. Dict dispatch over factories.
5. **Explicit over implicit.** Every module has a calling spec (inputs, outputs, side effects).
6. **Deterministic logic is sealed.** Tested tools, never AI-regenerated.
7. **Probabilistic logic is flexible.** LLM's domain.

## Implementation Workflow

### Step 1: Load and Validate Plan

1. Read the plan file from `docs/plans/PLAN_<name>.md`
2. Verify the plan has:
   - Phase 0 with target file tree, calling specs, responsibility mapping
   - Phase 1-N with RED/GREEN/REFACTOR tasks
   - Quality gates per phase
   - Consolidated checklist at the end
3. Identify current progress — find the first uncompleted phase
4. Announce: "Implementing PLAN_<name>.md. Starting at Phase {N}."

### Step 2: Execute Phase 0 (if not complete)

Phase 0 is architecture-only — no code written. Validate:
- [ ] Target file tree is defined with LOC estimates
- [ ] All files <= 800 LOC
- [ ] Calling specs defined for every module
- [ ] Responsibility mapping complete
- [ ] Applicable LOD patterns identified
- Check off Phase 0 items in the plan document
- Update plan status

If Phase 0 is already complete, skip to Phase 1.

### Step 3: Execute Phase 1-N (Implementation Loop)

For each implementation phase, follow this exact sequence:

#### 3a. RED Phase — Write Failing Tests First

1. Read the phase's RED tasks from the plan
2. Create test files at the paths specified (from Phase 0 tree)
3. Write test cases covering:
   - Happy path scenarios
   - Edge cases listed in the plan
   - Error conditions
4. **Run tests — verify they FAIL**
   ```bash
   [project test command]
   ```
5. If tests pass unexpectedly, the test is wrong — fix the test to properly assert the not-yet-implemented behavior
6. Check off RED task checkboxes in the plan

#### 3b. GREEN Phase — Minimal Implementation

1. Read the phase's GREEN tasks from the plan
2. Implement at the file paths specified (from Phase 0 tree)
3. Write **minimal code** to make tests pass — no extras
4. Every new module MUST start with a calling spec:
   ```
   """ModuleName -- Brief description.

   CALLING SPEC:
       result = function_name(
           param: Type,
       ) -> ReturnType

       Side effects: None | [describe]
       Deterministic: Yes | No
   """
   ```
5. **Run tests — verify they PASS**
6. If tests fail, fix implementation (not tests) until green
7. Check off GREEN task checkboxes in the plan

#### 3c. REFACTOR Phase — Clean Up

1. Read the phase's REFACTOR tasks from the plan
2. Improve code quality while keeping tests green:
   - Remove duplication
   - Improve naming
   - Extract pure functions where methods don't need `self`/`this` (Rule 3)
   - Ensure calling specs are accurate and complete (Rule 5)
   - Verify no file exceeds 800 LOC (Rule 1)
3. **Run tests after each change — must stay green**
4. Check off REFACTOR task checkboxes in the plan

#### 3d. Quality Gate Validation

After REFACTOR, run the full quality gate before proceeding:

**TDD Compliance:**
```bash
# Run tests
[project test command]

# Check coverage
[project coverage command]
```
- Verify: tests pass, coverage >= 80% business logic, no flaky tests

**LOD Conformance — validate every file touched in this phase:**
```bash
# Check file sizes (all must be <= 800 LOC)
wc -l [files created/modified in this phase]
```
- [ ] Files match Phase 0 target tree
- [ ] No file exceeds 800 LOC
- [ ] Each file has one responsibility
- [ ] Calling specs present on all module boundaries
- [ ] Pure functions used where no instance state needed
- [ ] No inheritance >1 level, no factory patterns

**Build & Code Quality:**
```bash
[project build command]
[project lint command]
[project type-check command]
```

**If any gate item fails:**
1. Fix the violation
2. Re-run tests (must still pass)
3. Re-validate the failed gate item
4. Do NOT proceed to next phase until all items pass

**If all gate items pass:**
1. Check off all quality gate checkboxes in the plan
2. Update the plan's phase status to Complete
3. Update "Last Updated" date
4. Add any learnings to the Notes section
5. Proceed to next phase

### Step 4: Consolidated Checklist (after all phases)

After the last implementation phase completes:

1. Run the consolidated LOD + TDD compliance checklist at the end of the plan
2. For each item, provide evidence:
   - LOD Hard Rules: run `wc -l` on all files, verify calling specs exist, check inheritance depth
   - LOD Patterns: verify each applicable pattern was correctly implemented
   - TDD: confirm all tests pass with coverage report
   - Code Quality: run linter, type checker, formatter
3. Check off each item with evidence
4. Update plan status to **Complete**

### Step 5: Summary Report

After completing all phases:

```
## Implementation Complete

**Plan**: PLAN_<name>.md
**Phases completed**: {N}
**Files created/modified**: [list]
**Test coverage**: X%
**LOD violations found and fixed**: [count]
**Time**: [if tracked]

### LOD Compliance
- Hard Rules: 7/7 passing
- Patterns applied: [list applicable patterns and their status]

### Next Steps
- [any follow-up items from the Notes section]
```

## Error Recovery

### Test Won't Pass (GREEN phase stuck)
1. Re-read the test — is it testing the right behavior?
2. Re-read the Phase 0 calling spec — does implementation match the spec?
3. Check if a dependency from a previous phase is missing
4. If the plan's task description is ambiguous, implement the simplest interpretation

### LOD Violation at Gate
1. **File too long (>800 LOC)**: Split using Radical Fragmentation — identify responsibilities and extract to separate files. Update Phase 0 tree if needed (document the deviation in Notes).
2. **Mixed responsibilities**: Extract the secondary concern to its own file with its own calling spec.
3. **Missing calling spec**: Add one at the top of the module following the spec format.
4. **Deep inheritance**: Flatten to composition or dict dispatch.
5. **Impure method that should be pure**: Extract to standalone function in a tools/ file.

### Plan Deviation
If implementation reveals the Phase 0 architecture needs adjustment:
1. Document the deviation in the plan's Notes section with justification
2. Update the Phase 0 target file tree to reflect reality
3. Verify the updated architecture still satisfies all 7 Hard Rules
4. Continue implementation with the updated plan

## Execution Principles

- **Follow the plan literally.** Don't add features, refactor beyond scope, or "improve" adjacent code.
- **One phase at a time.** Complete Phase N fully before starting Phase N+1.
- **Tests are the truth.** If tests pass and gates pass, the phase is done.
- **LOD violations are blockers.** Never defer a violation to "fix later."
- **Update the plan as you go.** Check boxes, update status, add notes. The plan is the living record.
- **Minimal code in GREEN.** Write the least code to pass tests. Elegance comes in REFACTOR.
- **Calling specs first.** Before implementing a module, write its calling spec. The spec IS the interface contract.
