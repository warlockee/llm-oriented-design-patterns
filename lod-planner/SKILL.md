# LOD Feature Planner

## What is LOD?

**LLM-Oriented Design (LOD)** is a set of architectural patterns that optimize code for AI comprehension and modification. The core insight: code that will be read, modified, and assembled by LLMs must be structured differently from code written only for humans.

**Why it matters**: LLMs have a finite intelligence window. A 2,000-line file forces the model to load irrelevant context. Mixed responsibilities increase hallucination risk. Implicit interfaces require reading full implementations. LOD solves these by enforcing small files, single responsibilities, explicit specs, and clean separation of deterministic (sealed) vs probabilistic (flexible) logic.

**LOD consists of**:
- **7 Hard Rules** -- always enforced, non-negotiable constraints (max 800 LOC, single responsibility, pure functions, flat hierarchy, calling specs, sealed deterministic logic, flexible probabilistic logic)
- **9 Design Patterns** -- situational methodologies applied when relevant (fragmentation, registries, toolification, orchestrators, schema separation, etc.)

## Purpose

Generate structured, phase-based feature plans where LOD patterns are enforced as hard architectural constraints. Every plan starts with Phase 0 (architecture decomposition) before any implementation, ensuring code is optimized for LLM comprehension from the start.

**Use lod-planner** when the codebase will be primarily read, modified, or assembled by LLMs, or when the feature involves 3+ new files or modifies modules over 300 LOC.

**Skip this skill** when LOD constraints are not desired, the change is small and self-contained (1-2 files, under 200 LOC), or the codebase is human-only maintained.

## LOD Hard Rules (ALWAYS ENFORCED)

These 7 rules are non-negotiable. Every plan must satisfy all of them. They are validated in full at Phase 0 gate and the consolidated checklist. Each Phase 1-N gate includes a single LOD conformance check referencing the Phase 0 architecture.

1. **Max 800 LOC per file.** Every file must fit within the model's optimal intelligence window. If a file exceeds this, split it.
2. **One file, one responsibility.** Never mix schema definition with sync logic, or core algorithms with checkpoint I/O.
3. **Pure functions over methods.** If logic doesn't need `self`/`this`, extract it into a standalone function.
4. **Flat over deep.** No inheritance chains beyond 1 level. No factory-of-factory patterns. A dict dispatch is better than a Strategy pattern.
5. **Explicit over implicit.** Every module starts with a calling spec -- inputs, outputs, side effects -- so the LLM never needs to read the implementation to use it.
6. **Deterministic logic is sealed.** Math formulas, file I/O, tensor ops, config validation -- these are tools. They are human-written, tested, and never regenerated by AI.
7. **Probabilistic logic is flexible.** Routing decisions, parameter selection, variant choice -- these are the LLM's domain.

## LOD Design Patterns (SITUATIONAL)

Apply these 9 patterns only when the feature's characteristics trigger them. Use the detection guide below to decide which apply.

**How patterns relate to Hard Rules**: Some patterns extend or amplify Hard Rules. The Hard Rule sets the minimum bar (always enforced). The pattern adds structured methodology when the feature needs it:
- **Pattern 1 (Radical Fragmentation)** amplifies Rules 1-2: Rules enforce the ceiling (800 LOC, single responsibility). Pattern 1 adds proactive decomposition strategy when splitting is needed — how to fragment, where to cut boundaries.
- **Pattern 2 (Calling Specs)** amplifies Rule 5: Rule 5 requires every module has a spec. Pattern 2 adds the recursive compression design — organizing specs as a hierarchy of black boxes across abstraction levels.
- **Pattern 8 (Dict Dispatch)** amplifies Rule 4: Rule 4 bans deep inheritance and factory patterns. Pattern 8 applies when there IS routing logic — providing the dict-based alternative.

When a pattern's skip condition matches, the Hard Rule still applies — but the pattern's extended methodology is not needed.

### Pattern 1: Radical Fragmentation

Split monolithic classes into single-responsibility micro-files.

**Anti-pattern** -- god class with 7 responsibilities (665 LOC, must read all to modify one concern):
```python
class BigService:
    def __init__(self, ...):        # engine setup
    def load_model(self):           # model I/O
    def compute_distance(self, ...): # math utility
    def compute_loss(self, ...):    # 3 algorithm variants behind if-else
    def train_step(self, ...):      # training orchestration
    def merge_weights(self, ...):   # weight merging (duplicated elsewhere)
    def save_checkpoint(self, ...): # file I/O (2 code paths)
```

**Pattern** -- shatter into focused files:
```
feature/
  losses/
    variant_a.py    # 67 LOC -- one variant per file
    variant_b.py    # 75 LOC
  tools/
    weight_merge.py # 61 LOC -- pure function, no class state
    checkpoint.py   # 84 LOC -- save/gather/config extraction
  service.py        # 200 LOC -- orchestration only
```

**Principle**: If a class has N responsibilities, modifying responsibility #3 requires loading all N. After splitting, load only #3.

### Pattern 2: Calling Specs as Black Boxes

Every module gets a calling spec at the top. The LLM reads the spec (~20-30 tokens) instead of the implementation (~500+ tokens).

```python
"""NotificationSender -- Dispatch notifications to registered channels.

CALLING SPEC:
    result = send_notification(
        notification=Notification,
        channel="email" | "sms" | "push",
        registry=dict[str, Callable],
    ) -> SendResult:
        SendResult: success (bool), channel (str), error (str | None)

    Side effects: External API call via channel implementation
    Deterministic: No
"""
```

**Recursive compression** -- at each level, read only the spec from below:
```
Level 0: channel_fn spec    -> "result = channel_fn(notification)"    ~20 tokens
Level 1: sender spec        -> "result = send(notif, channel)"       ~20 tokens
Level 2: orchestrator spec  -> "outcome = orchestrate(request)"      ~15 tokens
Total: ~55 tokens to understand pipeline (vs ~5,000 tokens reading source)
```

### Pattern 3: Variant Functions as Registry

When multiple implementations share the same interface, use a registry of pure functions with identical signatures. Dispatch via dict, not if-else.

**Anti-pattern**:
```python
def compute(self, data):
    if self.variant == "A":   # 30 LOC
    elif self.variant == "B": # 25 LOC
    elif self.variant == "C": # 20 LOC
```

**Pattern**:
```python
# variants/__init__.py
REGISTRY = {"A": compute_a, "B": compute_b, "C": compute_c}

# variants/variant_b.py -- single file, single responsibility
def compute_b(data, config_a, config_b):
    """Variant B: brief description."""
    return result, metrics_dict
```

Add a new variant = add one file + one registry entry. Zero risk to existing variants.

### Pattern 4: Toolification

Extract deterministic logic into standalone, tested, pure-function tools. If two classes share the same utility, it must be a tool, not a duplicated method.

```python
# tools/weight_merge.py
def merge_weights(state_dict, alpha, rank):
    """Merge adapter weights into base weights.

    TOOL CONTRACT:
        Input:  state_dict with base + adapter weights, scaling params
        Output: new state_dict with merged weights, adapter keys removed
        Side effects: None
        Deterministic: same input -> same output
    """
    return new_state_dict
```

Every tool contract states: inputs/outputs, deterministic guarantee, no side effects, independently testable.

### Pattern 5: Orchestrator Files as Recipes

The top-level entry point is a slim orchestrator (<200 LOC) that reads like a recipe. All logic lives in the phases it calls.

```python
# main.py -- ~120 LOC, pure orchestration
from myapp.setup import create_engines
from myapp.phases import run_processing, run_optimization, save_and_refresh

def main(config_file):
    config = load_config(config_file)
    engines = create_engines(config)
    for epoch in range(config.num_epochs):
        results = run_processing(engines, data, epoch)   # Phase 1
        metrics = run_optimization(engines, results)      # Phase 2
        save_and_refresh(engines, config, epoch)          # Phase 3
```

Each phase is a separate file (60-100 LOC). The orchestrator shows the full pipeline in ~40 LOC.

### Pattern 6: Schema Separated from Logic

Data definitions (what fields exist) must be separate from behavior (what to do with them).

**Anti-pattern** -- 483 LOC file mixing schema + logic + I/O:
```python
class DatabaseConfig(BaseModel): ...  # schema
class AppConfig(BaseModel):           # schema
    def sync_settings(self, env): ... # logic
def load_config(path, env): ...       # I/O
```

**Pattern**:
```
configs/
  schema.py    # Pydantic models only, zero logic
  sync.py      # Config synchronization / derivation logic
  loader.py    # File I/O + verification
```

### Pattern 7: Zero-Hallucination Contracts

Use validators to prevent invalid configurations. Strip the LLM of improvisational freedom on infrastructure parameters.

```python
class WorkerConfig(BaseModel):
    model_config = ConfigDict(extra='forbid')  # reject unknown fields
    num_workers: int = Field(default=8, ge=1, le=64)
    timeout_sec: int = Field(default=30, gt=0, le=3600)
    memory_fraction: float = Field(default=0.5, gt=0.0, le=0.95)
```

- `extra='forbid'` prevents hallucinated field names
- `Field(ge=1)` prevents nonsensical values
- Validation errors fire immediately -- no silent failures downstream

### Pattern 8: Dict Dispatch over Factory Patterns

Use plain dicts for routing. No AbstractFactory, no StrategyProvider, no RegistryManager.

```python
ALGORITHMS = {
    "variant_a": ClassA,
    "variant_b": ClassA,
    "variant_c": ClassB,
}
alg = ALGORITHMS[config.algorithm_name.lower()]
```

An LLM reads the dict and immediately knows all valid options. No indirection to trace.

### Pattern 9: Feedback Loops with Measurable Outcomes

Each processing phase returns a structured result. Downstream phases can inspect and react. Never log metrics without acting on anomalies.

```python
for step in range(n_steps):
    metrics = compute(batch)
    if math.isnan(metrics['loss']):
        logger.error("NaN loss at step %d -- halting", step)
        break
    if metrics['grad_norm'] > 100.0:
        logger.warning("grad_norm=%.1f -- potential instability", metrics['grad_norm'])
```

Design feedback at multiple tiers:
```
MICRO (per step):   NaN detection, gradient explosion    -> halt or warn
MACRO (per epoch):  Stagnation, divergence trends        -> suggest config changes
SYSTEM (per run):   OOM, timeout, crash                  -> auto-retry with adjusted params
```

## Pattern Relevance Detection Guide

For each pattern, check the trigger condition. If it matches, include the pattern in Phase 0. If the skip condition matches, do not apply.

| Pattern | Trigger (APPLY when...) | Skip (does NOT apply when...) |
|---------|------------------------|-------------------------------|
| 1. Radical Fragmentation | Feature would create or modify files that exceed 200 LOC without splitting (Rule 1-2 set ceiling; this pattern guides HOW to split) | Feature is a single utility under 100 LOC |
| 2. Calling Specs | Feature creates 2+ new modules with cross-module dependencies (Rule 5 requires specs; this pattern adds recursive black-box design) | Feature modifies only internal implementation of one module |
| 3. Variant Registry | 2+ implementations of same interface, OR existing if-else/switch dispatching on type/variant | There is genuinely only one implementation with no foreseeable variants |
| 4. Toolification | Deterministic logic is shared across 2+ components, OR duplicated methods exist | All logic is unique to one component and inherently stateful |
| 5. Orchestrator Recipes | Feature has a top-level entry point coordinating 3+ phases or steps | Feature is a leaf-level utility with no coordination role |
| 6. Schema Separation | Feature defines data structures AND behavior on those structures in the same scope | Feature only defines behavior using existing schemas, or only defines schemas |
| 7. Zero-Hallucination Contracts | Feature uses configuration with numeric, enum, or bounded params | Feature has no user-configurable parameters |
| 8. Dict Dispatch | Feature routes to different implementations based on string/enum keys (Rule 4 bans factories; this pattern provides the dict alternative) | Only one code path exists, no branching by type |
| 9. Feedback Loops | Feature has iterative processing that produces metrics or status signals | Feature is a one-shot operation with no iteration |

## Planning Workflow

### Step 1: Requirements Analysis
1. Read relevant files to understand codebase architecture
2. Identify dependencies and integration points
3. Assess complexity and risks
4. Determine appropriate scope (small/medium/large)

### Step 2: LOD Architecture Decomposition (Phase 0)
1. Analyze feature requirements and codebase context for LOD architecture
2. Produce target file tree with LOC estimates per file (all <= 800 LOC)
3. Define module boundaries and calling specs (inputs, outputs, side effects)
4. Map responsibilities to files (one file, one responsibility)
5. Run each pattern through the detection guide -- list applicable patterns with justification
6. Verify the proposed architecture satisfies all 7 Hard Rules

### Step 3: TDD Phase Breakdown (Phase 1-N)
Break the feature into 3-7 implementation phases where each phase:
- **Test-First**: Write tests BEFORE implementation (Red-Green-Refactor)
- Delivers working, testable functionality
- Takes 1-4 hours maximum
- **References Phase 0 file tree**: Every task's file reference must map to a file in Phase 0's target architecture
- Has LOD conformance check in its quality gate
- Can be rolled back independently
- Has clear success criteria

### Step 4: Consolidated Checklist Generation
Merge LOD compliance + TDD compliance + Code Quality into a final self-verification checklist at the end of the plan.

### Step 5: User Approval
**CRITICAL**: Use AskUserQuestion to get explicit approval before creating the plan document.

Ask:
- "Does this LOD architecture decomposition make sense for your codebase?"
- "Any concerns about the proposed file structure or module boundaries?"
- "Should I proceed with creating the plan document?"

Only create the plan document after user confirms approval.

### Step 6: Document Generation
1. Create `docs/plans/` directory if not exists
2. Generate plan document at `docs/plans/PLAN_<feature-name>.md` using plan-template.md with all checkboxes unchecked
3. Inform user of plan location and next steps

## Quality Gate Standards

Each phase quality gate includes:

**Phase 0 Gate** (full LOD architectural validation):
- All 7 Hard Rules validated against proposed architecture
- Applicable patterns identified with justification
- Target file tree with LOC estimates

**Phase 1-N Gates** (TDD + LOD conformance):
- Full TDD compliance (Red-Green-Refactor cycle)
- LOD conformance: `- [ ] Phase output conforms to Phase 0 architecture (files match target tree, no file exceeds 800 LOC)`
- Build, test, code quality, security, documentation checks

**Consolidated Checklist** (final verification):
- Full 7-item LOD Hard Rules verification
- Applicable LOD Pattern verification (only patterns from Phase 0)
- TDD compliance summary
- Code quality summary

## Risk Assessment

Identify and document:
- **Technical Risks**: API changes, performance issues, data migration, LOD pattern misapplication
- **Dependency Risks**: External library updates, third-party service availability
- **Timeline Risks**: Complexity unknowns, blocking dependencies
- **Quality Risks**: Test coverage gaps, regression potential, LOD violations discovered late

For each risk, specify:
- Probability: Low/Medium/High
- Impact: Low/Medium/High
- Mitigation Strategy: Specific action steps

## Rollback Strategy

For each phase, document how to revert changes if issues arise.
Consider:
- What code changes need to be undone (reference Phase 0 file tree)
- Database migrations to reverse (if applicable)
- Configuration changes to restore
- Dependencies to remove

## Test Specification Guidelines

### Test-First Development Workflow

**For Each Feature Component**:
1. **Specify Test Cases** (before writing ANY code)
   - What inputs will be tested?
   - What outputs are expected?
   - What edge cases must be handled?
   - What error conditions should be tested?

2. **Write Tests** (Red Phase)
   - Write tests that WILL fail
   - Verify tests fail for the right reason
   - Run tests to confirm failure

3. **Implement Code** (Green Phase)
   - Write minimal code to make tests pass
   - Run tests frequently (every 2-5 minutes)
   - Stop when all tests pass
   - No additional functionality beyond tests

4. **Refactor** (Blue Phase)
   - Improve code quality while tests remain green
   - Extract duplicated logic
   - Improve naming and structure
   - Add calling specs to module headers (LOD Rule 5)
   - Run tests after each refactoring step

### Test Types

**Unit Tests**:
- **Target**: Individual functions, methods, classes
- **Dependencies**: None or mocked/stubbed
- **Speed**: Fast (<100ms per test)
- **Isolation**: Complete isolation from external systems
- **Coverage**: >= 80% of business logic

**Integration Tests**:
- **Target**: Interaction between components/modules
- **Dependencies**: May use real dependencies
- **Speed**: Moderate (<1s per test)
- **Isolation**: Tests component boundaries
- **Coverage**: Critical integration points

**End-to-End (E2E) Tests**:
- **Target**: Complete user workflows
- **Dependencies**: Real or near-real environment
- **Speed**: Slow (seconds to minutes)
- **Isolation**: Full system integration
- **Coverage**: Critical user journeys

### Test Coverage Calculation

**Coverage Thresholds** (adjust for your project):
- **Business Logic**: >= 90% (critical code paths)
- **Data Access Layer**: >= 80% (repositories, DAOs)
- **API/Controller Layer**: >= 70% (endpoints)
- **UI/Presentation**: Integration tests preferred over coverage

**Coverage Commands by Ecosystem**:
```bash
# JavaScript/TypeScript
jest --coverage

# Python
pytest --cov=src --cov-report=html

# Go
go test -cover ./...

# Rust
cargo tarpaulin --out Html
```

### Common Test Patterns

**Arrange-Act-Assert (AAA)**:
```
test 'description of behavior':
  // Arrange: Set up test data and dependencies
  // Act: Execute the behavior being tested
  // Assert: Verify expected outcome
```

**Mocking/Stubbing Dependencies**:
```
test 'component should call dependency':
  // Create mock/stub
  // Configure mock behavior
  // Execute and verify
```

### Test Documentation in Plan

**In each phase, specify**:
1. **Test File Location**: Exact path where tests will be written (from Phase 0 tree)
2. **Test Scenarios**: List of specific test cases
3. **Expected Failures**: What error should tests show initially?
4. **Coverage Target**: Percentage for this phase
5. **Dependencies to Mock**: What needs mocking/stubbing?
6. **Test Data**: What fixtures/factories are needed?

## Phase Sizing Guidelines

**Small Scope** (2-3 phases, 3-6 hours total):
- Single component or simple feature, 3-5 new files
- Example: Add a new API endpoint with validation and tests

**Medium Scope** (4-5 phases, 8-15 hours total):
- Multiple components, 6-12 new files
- Example: User authentication system with schema, logic, and API layers

**Large Scope** (6-7 phases, 15-25 hours total):
- Complex feature spanning multiple areas, 12+ new files
- Example: AI-powered search with embeddings, indexing, and query pipeline
